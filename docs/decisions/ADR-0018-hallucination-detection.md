# ADR-0018: Hallucination Detection and Fact Verification System

## Status

Accepted

## Context

The agentic CI pipelines produce structured output documents — RFE assessments, security reviews, strategy documents — that contain factual claims about Red Hat products, architectures, Jira issues, codebases, and security properties. These claims are generated by LLMs and may be hallucinated: invented Jira issue details, nonexistent API endpoints, incorrect architectural claims, fabricated security vulnerabilities, or misattributed code behaviors.

Currently there is no systematic way to identify, verify, or triage potentially hallucinated content in pipeline outputs. The Observatory collects these artifacts but only displays them as raw files.

### Reference Materials Studied

- **Exa Hallucination Detector** — open-source three-stage pipeline: extract claims → web search for evidence → LLM-as-judge verdict
- **Claimify** (ACL 2025, Metropolitansky & Larson) — systematic factual claim extraction with decontextualization and element-level coverage
- **VeriTrail** (ICLR 2026, Metropolitansky & Larson) — closed-domain hallucination detection with traceability through multi-generative-step processes; uses Fully Supported / Not Fully Supported / Inconclusive verdict taxonomy
- **Semantic Entropy** (Nature 2024, Farquhar et al.) — intrinsic uncertainty detection via sampling + semantic clustering, no external sources needed
- **Datadog LLM-as-Judge** — rubric-based disagreement detection (contradictions, unsupported claims, agreements)
- **Braintrust survey** — landscape overview of hallucination detection tools and integration patterns

### What Our Pipelines Produce

| Pipeline | Output Type | Hallucination Risk |
|----------|------------|-------------------|
| RFE Assessor | Rubric-scored assessments with specific claims about Jira issue content | Claims about what the RFE says, customer impact, scope assessment |
| RFE Autofixer | Rewritten RFE descriptions with business justification | Fabricated customer names, revenue figures, market data |
| Strat Pipeline | Strategy documents with architecture and dependency claims | Incorrect component relationships, fabricated API surfaces |
| Security Reviews | Risk assessments with architecture queries and vulnerability claims | False vulnerability claims, incorrect architecture assertions, made-up CVEs |
| Epic Decomposer | Feature/story breakdowns with scope and dependency claims | Incorrect dependency chains, fabricated acceptance criteria |

### The Closed-Domain Advantage

Our use case is primarily **closed-domain**: the agents operate on specific Jira issues, architecture docs, and codebases that we have access to. This means we can verify claims against the source material (the Jira ticket text, the actual codebase, the architecture query results) rather than relying on web search. VeriTrail's approach is directly applicable.

## Decision

Build a hallucination detection system as a first-class Observatory feature with its own page at `/hallucinations` in the sidebar. The system follows a three-stage pipeline adapted from the reference materials:

### Stage 1: Claim Extraction

A batch ingestion script (`scripts/extract-claims.py`) processes artifact files from `./var/artifacts/` and extracts verifiable factual claims.

**Database table:**

```sql
CREATE TABLE claims (
    id INTEGER PRIMARY KEY,
    pipeline_run_id INTEGER REFERENCES pipeline_runs(id) ON DELETE CASCADE,
    pipeline_slug TEXT NOT NULL,
    source_file TEXT NOT NULL,
    claim_text TEXT NOT NULL,
    original_text TEXT,
    claim_type TEXT,           -- 'factual', 'architectural', 'security', 'scope', 'attribution'
    context TEXT,              -- surrounding text for disambiguation
    extracted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Extraction approach:**
- Use LLM-based decomposition (following Claimify/Exa pattern) with a prompt tuned for our artifact types
- Extract atomic, independently verifiable statements
- Classify each claim by type (factual, architectural, security, scope, attribution)
- Preserve the original text span for traceability

### Stage 2: Claim Verification

A verification script (`scripts/verify-claims.py`) evaluates extracted claims against available evidence sources.

**Verification strategies by claim type:**

| Claim Type | Evidence Source | Method |
|------------|---------------|--------|
| Jira claims ("RHAIRFE-123 says...") | Jira API | Fetch actual issue, compare claimed content |
| Architecture claims ("component X depends on Y") | `arch-query` results in artifacts, codebase | Compare against architecture query data already captured in security review artifacts |
| Code claims ("function X does Y") | Source repos in `./var/definitions/` | Grep/search the actual codebase |
| Security claims ("CVE-XXXX affects...") | Vulnerability databases, SBOM data | Cross-reference with existing SBOM/vuln tables |
| Scope/attribution claims | Jira API, git history | Verify attributions against actual data |

**Database table:**

```sql
CREATE TABLE claim_verdicts (
    id INTEGER PRIMARY KEY,
    claim_id INTEGER REFERENCES claims(id) ON DELETE CASCADE,
    verdict TEXT NOT NULL,       -- 'supported', 'refuted', 'insufficient', 'inconclusive'
    confidence INTEGER,          -- 0-100
    evidence_summary TEXT,
    evidence_source TEXT,        -- 'jira', 'codebase', 'architecture', 'web', 'llm-judge'
    evidence_detail TEXT,        -- raw evidence text
    verified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Verdict taxonomy** (following VeriTrail):
- **Supported**: Evidence strongly implies the claim is true
- **Refuted**: Evidence contradicts the claim
- **Insufficient**: No evidence found to evaluate
- **Inconclusive**: Evidence is ambiguous or conflicting

### Stage 3: Triage UI

A dedicated `/hallucinations` page with:

**Dashboard view:**
- Summary cards: total claims, verified count, refuted count, pending count
- Refutation rate by pipeline and claim type
- Trend chart of hallucination rate over time

**Claim browser:**
- Filterable table: pipeline, claim type, verdict, confidence
- Click to expand: full claim text, original context, evidence chain
- Triage actions: mark as reviewed, override verdict, add notes

**Pipeline integration:**
- Pipeline detail page shows a hallucination summary card (claims extracted, refuted count)
- Link from pipeline detail to filtered hallucination view

### Architecture

```
./var/artifacts/    ──→  extract-claims.py  ──→  claims table
                                                      │
                                                      ▼
                         verify-claims.py   ──→  claim_verdicts table
                              │
                              ├── Jira API (for Jira claims)
                              ├── ./var/definitions/ (for code claims)
                              ├── Architecture query data (for arch claims)
                              └── LLM-as-judge (for subjective claims)
                                                      │
                                                      ▼
                         /hallucinations page  ←──  REST API
```

### Implementation Phases

**Phase 1: Extraction + Storage**
- `claims` and `claim_verdicts` tables
- `scripts/extract-claims.py` — LLM-based claim extraction from artifact markdown files
- `make extract-claims` target
- Basic API endpoints: list claims, filter by pipeline/type/verdict

**Phase 2: Automated Verification**
- `scripts/verify-claims.py` — multi-strategy verification
- Verify claims against co-located source material in the artifact directories (no Jira API needed — source text is already captured in artifacts)
- Add codebase verification using `./var/definitions/`

**Phase 3: UI + Triage**
- `/hallucinations` page with dashboard + claim browser
- Sidebar navigation entry
- Pipeline detail integration

**Phase 4: Advanced**
- Confidence calibration across verification methods
- Hallucination rate tracking over time (are pipelines improving?)
- Feedback loop: refuted claims inform prompt engineering for the pipelines

## Consequences

Positive:
- Systematic quality assurance for AI-generated pipeline outputs
- Closed-domain verification is highly accurate — we have the ground truth data
- Traceability from claim → evidence → verdict enables debugging and prompt improvement
- Hallucination rate becomes a measurable quality metric per pipeline
- The triage UI enables human-in-the-loop review without reading every artifact

Negative:
- Claim extraction requires LLM calls — adds cost and API dependency
- False positives in claim extraction (extracting non-claims) add noise
- The system itself can hallucinate verdicts — needs calibration and human review
- Large artifact volumes need filtering (see Artifact File Selection below)

## Artifact File Selection

Not all artifact files should be processed. The extraction script applies these filters:

### Files to PROCESS (agent outputs with verifiable claims)

| Pipeline | File Pattern | Description |
|----------|-------------|-------------|
| strat-security-reviews | `*-security-review.md` | Final synthesized security review |
| strat-security-reviews | `*-reviewer-{1,2,3}.md` | Independent reviewer outputs |
| strat-security-reviews | `*-security-requirements.md` | Generated security requirements |
| strat-security-reviews | `*-threat-surface.md` | Extracted threat inventory |
| strat-pipeline | `RHAISTRAT-*` in strat-tasks/ | Generated strategy documents |
| strat-pipeline | `RHAISTRAT-*-review.md` in strat-reviews/ | Strategy reviews |

### Files to SKIP (ground truth / source inputs / duplicates)

| Pattern | Reason |
|---------|--------|
| `*-strat-text.md` | Source input — the original strategy text the agent reviews |
| `strat-originals/` and `strat-originals-old/` | Source RFE texts the agent works from |
| `STRAT-*` (without RHAISTRAT prefix) in strat-tasks/ | Legacy naming, superseded by RHAISTRAT-prefixed files |
| `ci-jobs/` subdirectories | Duplicates of data-repo files (one copy per pipeline run) |
| `README.md`, `strat-rubric.md` | Static metadata, not agent output |

### Pipeline scope

Initial extraction focuses on strat pipelines only (~800 files). RFE pipelines (93k files) are deferred — they're repetitive per-issue rubric applications and should be sampled rather than exhaustively processed.

### Cost estimate

- ~800 strat files × ~$0.01-0.03 per extraction = **$8-24 total**
- Uses Claude Sonnet via Vertex AI for extraction (cheaper than Opus, sufficient quality for claim decomposition)
