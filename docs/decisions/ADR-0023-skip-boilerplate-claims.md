# ADR-0023: Skip Boilerplate Claims in Extraction and Verification

## Status

Accepted

## Context

Analysis of the 32,645 extracted claims shows that the majority are boilerplate metadata copied from document frontmatter or rubric output. These claims are trivially true (the document literally contains the stated value) and consume API calls during both extraction and verification without catching real hallucinations.

### Top claims by cross-issue frequency

| Claim | Issues | Category |
|-------|--------|----------|
| The strategy status is Refined | 141 | metadata |
| The status of this strategy is Refined | 57 | metadata (duplicate phrasing) |
| The strategy priority is Major | 53 | metadata |
| The strategy received a scope score of 2 out of 2 | 51 | rubric score |
| The strategy received a feasibility score of 1 out of 2 | 40 | rubric score |
| The strategy was generated by an Agentic SDLC Pipeline | 33 | attribution boilerplate |
| The effort estimate is Medium, estimated at 3-5 sprints | 26 | effort estimate |
| The scope reviewer verdict is Approve | 14 | reviewer verdict |
| The strategy's needs_attention flag is set to false | 11 | flag value |

These represent ~60% of all claims but ~0% of hallucination risk. They are direct transcriptions of document fields, not reasoning output.

### Claims worth verifying

The remaining ~40% are architectural, security, and relationship claims that involve agent reasoning:

| Claim | Issues | Why it matters |
|-------|--------|---------------|
| The gen-ai-ui plugin runs on port 8143 | 13 | Verifiable against arch-query |
| The Dashboard Team owns odh-dashboard | 17 | Ownership assertion |
| OGX was formerly known as Llama Stack | 11 | Naming fact |
| EvalHub uses a PostgreSQL database | 6 | Architectural dependency |
| RHAISTRAT-1576 has four dependency levels | 4 | Scope/complexity claim |

## Decision

Filter out boilerplate claims at two stages:

### 1. Extraction (extract-claims.py)

Update the extraction prompt to instruct the LLM to skip:
- Document metadata (status, priority, labels, dates, flags)
- Rubric scores and totals ("received a score of X out of Y")
- Reviewer verdicts ("the X reviewer verdict is Y")
- Effort estimates ("estimated at N sprints")
- Generator attribution ("generated by an Agentic SDLC Pipeline")
- Duplicate phrasings of the same metadata fact

### 2. Verification (verify-claims.py)

Add a pre-filter that deletes existing boilerplate claims from the database and skips them during future verification runs. No reason to keep trivially-true metadata — it inflates claim counts and adds noise to every query.

### Boilerplate patterns to skip

```python
BOILERPLATE_PATTERNS = [
    r"^The strategy (status|priority) is ",
    r"^The status of this strategy is ",
    r"^The priority of this strategy is ",
    r"strategy received a .* score of \d+ out of \d+",
    r"score is \d+ out of \d+",
    r"total score is \d+ out of \d+",
    r"generated by an Agentic SDLC Pipeline",
    r"effort estimate .* estimated at .* sprints?",
    r"reviewer verdict is ",
    r"overall recommendation is ",
    r"needs_attention flag is set to ",
    r"strategy has a status of ",
    r"strategy has not been revised since",
]
```

## Consequences

Positive:
- ~60% fewer API calls during extraction (future runs)
- ~60% fewer verification calls — focus budget on claims that matter
- Cleaner hallucination dashboard — boilerplate noise doesn't drown out real findings
- Faster verification runs

Negative:
- A boilerplate claim that IS wrong (e.g., score doesn't match the rubric table) would be missed
- The pattern list needs maintenance as new document formats appear
